# 原神千星奇域·综合指南 - 成果物收集表

## 1. 目的与适用范围

### 1.1 目的
本收集表用于规范AI智能体输出成果的格式与提交要求，确保所有成果物符合项目标准，便于管理和查阅。

### 1.2 适用范围
适用于原神千星奇域·综合指南网站爬取项目中所有AI智能体生成的成果物，包括：
- Markdown文档（常规文档和分段文档）
- 目录文件
- 自检报告
- 进度报告
- 其他项目相关文件

## 2. 成果物分类与命名规范

### 2.1 成果物分类

| 分类 | 子分类 | 描述 | 示例 |

### 2.2 命名规范

#### 2.2.1 常规文档命名
- **格式**：`中文标题.md`
- **示例**：`读前须知.md`
- **要求**：
  - 标题与原始页面一致
  - 使用中文，清晰反映文档内容
  - 不含特殊字符（如? * : " < > |）
  - 长度不超过50字符

#### 2.2.2 分段文档命名
- **格式**：`原文件名_一级标题_二级标题.md`
- **示例**：`执行节点_一、通用节点_1.打印字符串.md`
- **要求**：
  - 原文件名为极长内容的主文件名
  - 一级标题为h1标签内容
  - 二级标题为h2标签内容
  - 标题间用下划线分隔
  - 不含特殊字符

#### 2.2.3 目录文件命名
- **格式**：`原文件名 - 目录.md`
- **示例**：`执行节点 - 目录.md`
- **要求**：
  - 原文件名为极长内容的主文件名
  - 使用" - 目录"作为后缀

#### 2.2.4 报告文件命名
- **自检报告**：`自检报告_YYYY-MM-DD_HH-MM-SS.md`
- **进度报告**：`进度报告_YYYY-MM-DD.md`
- **总结报告**：`项目总结报告.md`
- **要求**：
  - 包含报告类型和日期（自检报告需包含时间）
  - 使用下划线分隔
  - 日期格式为YYYY-MM-DD
  - 时间格式为HH-MM-SS

#### 2.2.5 配置文件命名
- **映射表**：`doc_id_map.json`（固定名称）
- **爬取日志**：`crawl_log.txt`（固定名称）

#### 2.2.6 图片资源命名
- **格式**：`images/哈希值.png`
- **示例**：`images/1234567890abcdef.png`
- **要求**：
  - 存储在images目录下
  - 使用图片URL的哈希值作为文件名
  - 避免命名冲突
  - 保留原始文件格式

## 3. 成果物格式要求

### 3.1 Markdown文档格式

#### 3.1.1 基本结构
```markdown
# 文档标题

## 一级章节

### 二级章节

段落内容...

- 列表项1
- 列表项2
- 列表项3

| 表头1 | 表头2 | 表头3 |
|-------|-------|-------|
| 内容1 | 内容2 | 内容3 |
| 内容4 | 内容5 | 内容6 |

```代码块
代码内容...
```
```

#### 3.1.2 格式规范
- 使用UTF-8编码
- 行尾使用LF换行符
- 不使用BOM头
- 每个段落单独一行
- 行宽控制在100字符以内
- 代码块使用 ``` 包裹
- 表格使用标准Markdown格式

### 3.2 JSON配置文件格式

- 使用UTF-8编码
- 格式化输出，缩进2个空格
- 键名使用双引号
- 字符串值使用双引号
- 无多余逗号

### 3.3 日志文件格式

- 使用UTF-8编码
- 每行一条记录
- 包含时间戳、级别、内容
- 时间戳格式：YYYY-MM-DD HH:MM:SS

## 5. 成果物提交流程

### 5.1 提交时间

| 成果物类型 | 提交时间 | 负责人 |
|------------|----------|--------|
| 常规文档 | 生成后立即提交 | AI智能体 |
| 分段文档 | 分段处理完成后提交 | AI智能体 |
| 目录文件 | 分段处理完成后提交 | AI智能体 |
| 自检报告 | 检查完成后立即提交 | AI智能体 |
| 进度报告 | 每天结束时提交 | AI智能体 |
| 总结报告 | 项目结束时提交 | AI智能体 |
| 配置文件 | 生成后立即提交 | AI智能体 |
| 图片资源 | 下载后立即提交 | AI智能体 |

### 5.2 提交方式

1. **自动提交**：AI智能体生成成果物后自动保存到指定目录
2. **手动提交**：特殊情况下，人工协助将成果物保存到指定目录
3. **提交验证**：AI智能体自动验证成果物格式和命名规范
4. **提交记录**：每次提交自动生成提交记录

### 5.3 提交验证

| 验证项 | 验证内容 | 验证方式 | 失败处理 |
|--------|----------|----------|----------|
| 命名规范 | 文件名是否符合命名规则 | 自动验证 | 重命名后重新提交 |
| 格式规范 | 文件格式是否符合要求 | 自动验证 | 重新生成后提交 |
| 存储位置 | 是否存储在指定目录 | 自动验证 | 移动到正确目录 |
| 完整性 | 文件内容是否完整 | 自动验证 | 重新生成后提交 |

## 7. 成果物检索与查询

### 7.1 检索方式

| 检索项 | 检索方式 | 适用场景 |
|--------|----------|----------|
| 文件名 | 直接通过文件名查找 | 已知文件名 |

## 8. 成果物版本管理

### 8.1 版本控制

- 使用文件系统的修改时间作为版本标识
- 重要文件的修改需记录修改日志
- 重大变更需备份原始文件

### 8.2 版本命名

对于需要多次修改的文件，采用以下命名方式：
- **格式**：`文件名_版本号.md`
- **示例**：`自检报告规范_v1.1.md`
- **版本号规则**：主版本号.次版本号
  - 主版本号：重大变更时递增
  - 次版本号：小修改或优化时递增

## 9. 成果物归档与清理

### 9.1 归档要求

- 项目结束后，所有成果物统一归档
- 归档格式：ZIP或TAR.GZ
- 归档命名：`项目名称_归档日期.zip`
- 归档内容包括所有成果物和配置文件

### 9.2 清理规则

- 临时文件在使用完毕后立即清理
- 测试文件在测试完成后清理
- 重复文件保留最新版本
- 已废弃的文件定期清理
