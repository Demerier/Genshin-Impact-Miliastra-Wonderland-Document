# 提交规范指南

## 1. 概述

本项目采用 [Conventional Commits](https://www.conventionalcommits.org/zh-hans/v1.0.0/) 规范，用于统一提交信息格式，便于自动生成 CHANGELOG 和版本管理。

## 2. 提交信息格式

```
<类型>(可选的范围): <描述>

[可选的正文]

[可选的页脚]
```

## 3. 类型

提交信息的类型必须是以下之一：

| 类型       | 描述                                                         |
|------------|--------------------------------------------------------------|
| feat       | 新增功能                                                     |
| fix        | 修复bug                                                      |
| docs       | 仅修改文档                                                   |
| style      | 不影响代码含义的修改（空白、格式、缺少分号等）               |
| refactor   | 既不修复bug也不新增功能的代码修改                           |
| test       | 添加或修改测试代码                                           |
| chore      | 不涉及源代码或测试代码的修改（构建过程、依赖管理等）         |
| perf       | 优化性能的代码修改                                           |
| ci         | 持续集成相关配置的修改                                       |
| revert     | 回滚之前的提交                                               |

## 4. 范围

范围是可选的，用于指定提交影响的模块或组件，例如：

```
feat(crawler): 添加新的爬取策略
fix(parser): 修复HTML解析错误
```

## 5. 描述

描述是对提交内容的简短总结，应该：

- 以动词开头，使用第一人称现在时（如"添加"而非"添加了"）
- 不超过50个字符
- 结尾不加句号

## 6. 正文

正文是对提交内容的详细描述，应该：

- 换行符分隔，每行不超过72个字符
- 解释"什么"和"为什么"，而不是"如何"（代码已经说明了如何实现）

## 7. 页脚

页脚用于关联issue或PR，例如：

```
Closes #123
Fixes #456
```

## 8. 示例

### 8.1 新增功能

```
feat: 添加用户认证功能

实现了基于JWT的用户认证系统，包括注册、登录和权限验证。

Closes #123
```

### 8.2 修复bug

```
fix(parser): 修复HTML解析错误

解决了特殊字符导致的解析失败问题。

Fixes #456
```

### 8.3 修改文档

```
docs: 更新README.md

添加了项目架构图和快速开始指南。
```

### 8.4 代码重构

```
refactor(crawler): 重构爬取逻辑

优化了爬取流程，提高了性能和可维护性。
```

## 9. 工具支持

可以使用以下工具来辅助遵循提交规范：

- [commitlint](https://commitlint.js.org/)：提交信息校验
- [cz-cli](https://github.com/commitizen/cz-cli)：交互式提交信息生成
- [standard-version](https://github.com/conventional-changelog/standard-version)：自动生成CHANGELOG和版本标签

## 10. 最佳实践

- 每个提交只包含一个逻辑变更
- 提交信息应该清晰、准确、完整
- 使用英语或中文编写提交信息（建议统一使用中文）
- 遵循语义化版本控制原则
